# K.7.8.2A.11 — Contrato JSON Mestre

## Objetivo

Definir o contrato canônico que organiza os dez componentes aprovados do pacote fictício **Saúde → Clínica de Exames**, preparando a futura publicação controlada no Supabase e a consulta pela API da Vercel.

## Artefatos

- `data/employee-simulations/health/medical-testing-clinic/master-package.json`
- `data/employee-simulations/health/medical-testing-clinic/master-package.schema.json`

## Modelo adotado

O GitHub permanece como fonte mestre. Os arquivos individuais continuam canônicos. Na etapa de publicação, o pacote será montado como payload autocontido e validado antes de ser gravado no catálogo operacional.

```txt
SOURCE_OF_TRUTH=GITHUB
ASSEMBLY_MODE=SELF_CONTAINED_PAYLOAD_AT_PUBLICATION
RUNTIME_CATALOG=SUPABASE
DELIVERY_LAYER=VERCEL_API
USER_STATE=SESSION_STORAGE
```

## Componentes obrigatórios

1. Perfil da clínica fictícia.
2. Questionário e respostas sugeridas.
3. Catálogo de exames e serviços.
4. Horários, canais e unidades.
5. Agendamentos e cancelamentos.
6. Pagamentos e convênios.
7. Perguntas frequentes.
8. Encaminhamento humano.
9. Cenários e respostas esperadas.
10. Prompt operacional completo.

A ausência de qualquer componente obrigatório bloqueia a geração completa.

## Inventário esperado

```txt
SERVICES_CATEGORIES=4
SERVICES=12
UNITS=2
CHANNELS=4
FAQ_ITEMS=24
HANDOFF_PRIORITIES=4
HANDOFF_QUEUES=9
HANDOFF_LIFECYCLE_STATES=8
TEST_SCENARIOS=24
PROMPT_SECTIONS=13
```

## Regras cruzadas

- Todos os componentes JSON devem declarar `fictional=true`.
- Os componentes devem usar o mesmo `profileId` quando esse campo existir.
- Referências de serviço, unidade, prioridade e fila devem apontar para itens existentes.
- O prompt não pode ampliar permissões ou informações além dos componentes publicados.
- Contradição ou ausência deve bloquear a geração e exigir revisão humana.

## Invariantes de segurança

```txt
REAL_CUSTOMER_DATA=NO
REAL_PATIENT_DATA=NO
REAL_RECORD_ACCESS=NO
REAL_BOOKING=NO
REAL_CANCELLATION=NO
REAL_PAYMENT=NO
REAL_INSURANCE_LOOKUP=NO
CLINICAL_ADVICE=NO
RESULT_INTERPRETATION=NO
EXTERNAL_ACTION=NO
HUMAN_CONTACT_GUARANTEE=NO
TIME_PROMISE=NO
```

## Contrato de runtime

Prioridade de fonte:

1. pacote publicado no Supabase;
2. fallback versionado incluído no build.

Estados:

```txt
READY
PARTIAL
NOT_READY
```

Sem Supabase, o sistema poderá usar o fallback versionado. Sem as duas fontes, deverá bloquear a geração. Recuperação parcial será permitida somente por allowlist.

## Personalização do usuário

As escolhas e edições do usuário permanecem em `sessionStorage`, limitadas à mesma aba e sem persistência no Supabase no MVP.

## Gate de publicação

Antes da etapa 12, serão exigidos:

- presença de todos os componentes;
- validação pelo JSON Schema;
- referências cruzadas válidas;
- inventário esperado confirmado;
- invariantes de segurança confirmadas;
- alinhamento do prompt;
- checksum SHA-256 do pacote montado.

## Escopo preservado

Esta etapa não publica no Supabase, não cria API da Vercel e não integra o pacote ao criador de prompt.

```txt
SUPABASE_CHANGE=NO
VERCEL_CHANGE=NO
PRODUCTION_CHANGED=NO
MERGE=BLOCKED
```
