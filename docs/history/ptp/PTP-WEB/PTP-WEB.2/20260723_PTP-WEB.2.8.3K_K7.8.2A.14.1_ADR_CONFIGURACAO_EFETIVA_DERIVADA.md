## 2. Contexto

A validação visual e conversacional do pacote Saúde → Clínica de Exames demonstrou que as personalizações do questionário são armazenadas, mas não materializadas integralmente nas telas e no prompt.

```txt
QUESTIONÁRIO=Sophia / Clínica Ciame / Formal e detalhado
TELAS=Clara / Clínica Aurora / profissional, claro e respeitoso
PROMPT=texto canônico original + seção final de personalizações
```

A referência à Clínica Ciame foi uma personalização autorizada pelo usuário. O defeito real é a existência de representações divergentes da mesma configuração.

## 3. Problema arquitetural

O sistema possui atualmente:

1. `PackageDocument` com respostas sugeridas e prompt canônico;
2. `PackageCustomization` com respostas editadas e omissões;
3. catálogo adaptado construído com respostas sugeridas;
4. configuração genérica do criador;
5. prompt canônico seguido de uma seção corretiva.

Esse desenho permite divergência entre questionário, cartões, resumo e prompt.

## 4. Alternativas avaliadas

### 4.1 Consultar o questionário diretamente em cada tela

**Decisão:** rejeitada. Cada consumidor teria de interpretar valores, opções, rótulos e dependências novamente.

### 4.2 Persistir um `AgentDraft` editável

**Decisão:** rejeitada. Criaria uma nova fonte capaz de ficar dessincronizada do `PackageCustomization`.

### 4.3 Substituição textual global no prompt

**Decisão:** rejeitada. É frágil, pode alterar conteúdo indevido e não resolve bindings semânticos.

### 4.4 Reescrever manualmente todo o prompt

**Decisão:** rejeitada. Existe risco elevado de perder regras operacionais, encaminhamentos ou proteções.

### 4.5 Configuração efetiva derivada + materialização por seções

**Decisão:** aprovada.

```txt
PackageDocument
+
PackageCustomization
↓
buildEffectiveAgentConfig()
↓
EffectiveAgentConfig imutável
├── projeção visual
├── resumo
├── prompt materializado
├── testes
└── hashes de rastreabilidade
```

## 5. Decisão central

```txt
FONTES_PERSISTIDAS=PackageDocument + PackageCustomization
PROJEÇÃO_DERIVADA=EffectiveAgentConfig
PERSISTIR_PROJEÇÃO=NO
EDITAR_PROJEÇÃO=NO
```

O campo `includeInPromptAs` será usado como binding real. Cada valor deverá distinguir:

```txt
rawValue
displayValue
promptInstruction
source=suggested|user_edited|generated
```

## 6. Contrato obrigatório dos bindings

A implementação deverá usar allowlist exata e cobertura mensurável.

```txt
TOTAL_PACKAGE_QUESTIONS=N
BOUND_QUESTIONS=N
OPTIONAL_OMITTED=M
UNBOUND_REQUIRED=0
UNKNOWN_BINDINGS=0
DUPLICATE_BINDINGS=0
BINDING_COVERAGE=100%
```

A função `setByPath()` deverá operar somente sobre propriedades próprias e bloquear:

```txt
__proto__
prototype
constructor
```

Regras adicionais:

```txt
PROTOTYPE_POLLUTION=BLOCK
OWN_PROPERTIES_ONLY=YES
EXACT_ALLOWLIST_ONLY=YES
MAX_PATH_DEPTH=4
```

## 7. Origem de `promptInstruction`

Nesta etapa, as instruções semânticas de opções serão fornecidas por um mapa versionado dentro do materializador específico da clínica.

```txt
PROMPT_INSTRUCTION_SOURCE=VERSIONED_CLINIC_MATERIALIZER_MAP
PROMPT_INSTRUCTION_MAP_VERSION=1
MISSING_PROMPT_INSTRUCTION=BLOCK
```

O schema universal de instruções será tratado apenas em etapa multinicho futura.

## 8. Hashes e canonicalização

Todos os hashes derivados deverão obedecer ao mesmo contrato:

```txt
HASH_ALGORITHM=SHA-256
SERIALIZATION=STABLE_JSON_SORTED_KEYS
ARRAY_ORDER=PRESERVED
UNICODE_NORMALIZATION=NFC
CANONICALIZATION_VERSION=1
TEMPORAL_FIELDS_EXCLUDED=YES
```

Campos temporais e identificadores de sessão não participam dos hashes.

## 9. Materialização do prompt

O modelo atual será removido:

```txt
prompt canônico
+
seção 14 de personalizações
```

Será adotada materialização por seções com contrato fail-closed:

```txt
EXPECTED_SECTION_COUNT=13
MISSING_SECTION=BLOCK
DUPLICATE_SECTION=BLOCK
UNEXPECTED_TOP_LEVEL_SECTION=BLOCK
SECTION_ORDER_CHANGED=BLOCK
EMPTY_SECTION=BLOCK
CANONICAL_SOURCE_HASH_MISMATCH=BLOCK
```

O parser deverá preservar listas, blocos de código, linhas em branco, títulos internos e Unicode. Somente títulos de primeiro nível previstos no contrato podem delimitar as treze seções.

## 10. Paridade semântica objetiva

A configuração padrão somente será aprovada quando comprovar:


## 11. Projeção visual

O modo pacote disponibilizará uma interface pequena de projeção. O criador genérico continuará funcionando sem dependência obrigatória do pacote.

```txt
PACKAGE_PROJECTION_AVAILABLE=usar valores efetivos
PACKAGE_PROJECTION_UNAVAILABLE=manter comportamento genérico
```

## 12. Conflitos

Conflitos de identidade serão bloqueantes.

```txt
employee.name=Sophia
employee.initialMessage=Olá. Sou Clara...
PROMPT_GENERATION=BLOCKED
```

Texto editado manualmente não será alterado silenciosamente.

## 13. Segurança

As invariantes permanecem no `PackageDocument` e são imutáveis:

```txt
administrativeOnly=true
realCustomerDataAllowed=false
realPatientDataAllowed=false
realRecordAccessAllowed=false
realBookingAllowed=false
realPaymentAllowed=false
clinicalAdviceAllowed=false
resultInterpretationAllowed=false
```

Nenhuma personalização pode remover, omitir ou reduzir essas proteções.

## 14. Estratégia de entrega

A implementação deverá ocorrer em PR empilhado e isolado:

```txt
BASE=ptp-web-2-workforce-k6-questionnaire-r2
HEAD_PROPOSTO=ptp-web-2-workforce-effective-config-r1
PR_FILHO=DRAFT
PRODUÇÃO=INALTERADA
```

O PR #9 não deve crescer diretamente com toda a remediação.

## 15. Governança de deploy

A integração GitHub → Vercel gera Preview automático para commits da branch monitorada.

```txt
MANUAL_DEPLOY=BLOCKED
PRODUCTION_DEPLOY=BLOCKED
AUTOMATIC_GIT_PREVIEW=ENABLED_AND_OBSERVED
```

Antes da implementação deverá ser decidida separadamente uma política para commits apenas documentais:

1. aceitar Preview automático;
2. configurar Ignore Build Step para `docs/**` e `reports/**`;
3. usar branch documental não monitorada.

Nenhuma dessas opções está autorizada por este ADR.
