## 2. Contexto

A validação visual e conversacional do pacote Saúde → Clínica de Exames demonstrou que as personalizações do questionário são armazenadas, mas não materializadas integralmente nas telas e no prompt.

Exemplo observado:

```txt
QUESTIONÁRIO=Sophia / Clínica Ciame / Formal e detalhado
TELAS=Clara / Clínica Aurora / profissional, claro e respeitoso
PROMPT=texto canônico original + seção final de personalizações
```

A referência à Clínica Ciame não foi contaminação externa. Ela foi uma personalização autorizada pelo usuário. O defeito real é a existência de representações divergentes da mesma configuração.

## 3. Problema arquitetural

O sistema possui atualmente:

1. `PackageDocument` com respostas sugeridas e prompt canônico;
2. `PackageCustomization` com respostas editadas e omissões;
3. catálogo adaptado construído com respostas sugeridas;
4. configuração genérica do criador;
5. prompt canônico seguido de uma seção corretiva.

Esse desenho permite divergência entre questionário, cartões, resumo e prompt.

## 4. Alternativas avaliadas

### 4.1 Consultar o questionário diretamente em cada tela

**Decisão:** rejeitada.

Motivo: cada consumidor teria de interpretar novamente valores sugeridos, editados, omitidos, opções, rótulos e dependências.

### 4.2 Persistir um `AgentDraft` editável

**Decisão:** rejeitada.

Motivo: criaria uma nova fonte de dados capaz de ficar dessincronizada do `PackageCustomization`.

### 4.3 Fazer substituição textual global no prompt

**Decisão:** rejeitada.

Motivo: substituições por texto são frágeis, podem alterar conteúdo indevido e não resolvem bindings semânticos.

### 4.4 Reescrever manualmente todo o prompt de 13 seções

**Decisão:** rejeitada.

Motivo: risco elevado de perder regras operacionais, encaminhamentos ou proteções.

### 4.5 Configuração efetiva derivada + materialização por seções

**Decisão:** aprovada.

Fluxo:

```txt
PackageDocument
+
PackageCustomization
↓
buildEffectiveAgentConfig()
↓
EffectiveAgentConfig imutável
├── projeção visual
├── resumo
├── prompt materializado
├── testes
└── hashes de rastreabilidade
```

## 5. Decisão

A configuração final será uma projeção determinística e imutável, calculada a partir das duas fontes persistidas:

```txt
FONTES_PERSISTIDAS=PackageDocument + PackageCustomization
PROJEÇÃO_DERIVADA=EffectiveAgentConfig
PERSISTIR_PROJEÇÃO=NO
EDITAR_PROJEÇÃO=NO
```

O campo `includeInPromptAs` será usado como binding real para caminhos como:

```txt
company.displayName
employee.name
employee.presentation
employee.initialMessage
communication.tone
communication.responseLength
operation.channels
knowledge.authorizedSources
output.responseStructure
```

Cada valor deverá distinguir:

```txt
rawValue
 displayValue
 promptInstruction
 source=suggested|user_edited|generated
```

## 6. Materialização do prompt

O modelo atual será removido:

```txt
prompt canônico
+
seção 14 de personalizações
```

Será adotada materialização por seções:

1. reconhecer a estrutura canônica;
2. preservar seções operacionais imutáveis;
3. substituir somente blocos afetados pelas personalizações;
4. validar resíduos e conflitos;
5. bloquear a geração quando a estrutura não puder ser reconhecida com segurança.

Não será criado ainda um compilador universal multinicho.

## 7. Projeção visual

O modo pacote disponibilizará uma interface pequena de projeção. O criador genérico continuará funcionando sem dependência obrigatória do pacote.

```txt
PACKAGE_PROJECTION_AVAILABLE=usar valores efetivos
PACKAGE_PROJECTION_UNAVAILABLE=manter comportamento genérico
```

## 8. Conflitos

Conflitos de identidade serão bloqueantes.

Exemplo:

```txt
employee.name=Sophia
employee.initialMessage=Olá. Sou Clara...
PROMPT_GENERATION=BLOCKED
```

Texto editado manualmente não será alterado silenciosamente. O usuário deverá revisar ou aceitar explicitamente uma sugestão regenerada.

## 9. Segurança

As invariantes permanecem no `PackageDocument` e são imutáveis nesta etapa:

```txt
administrativeOnly=true
realCustomerDataAllowed=false
realPatientDataAllowed=false
realRecordAccessAllowed=false
realBookingAllowed=false
realPaymentAllowed=false
clinicalAdviceAllowed=false
resultInterpretationAllowed=false
```

Nenhuma personalização pode remover, omitir ou reduzir essas proteções.

## 10. Estratégia de entrega

A implementação deverá ocorrer em PR empilhado e isolado:

```txt
BASE=ptp-web-2-workforce-k6-questionnaire-r2
HEAD_PROPOSTO=ptp-web-2-workforce-effective-config-r1
PR_FILHO=DRAFT
PRODUÇÃO=INALTERADA
```

O PR #9 não deve crescer diretamente com toda a remediação.

## 12. Consequências

### Positivas

- uma única interpretação das personalizações;
- telas e prompt derivados do mesmo contrato;
- redução do risco de identidade residual;
- preservação do criador genérico;
- preparação para futuros pacotes sem generalização prematura.

### Custos

- novo contrato de bindings;
- projeção visual opcional;
- materializador específico da clínica;
- novos testes de determinismo e conflito.
