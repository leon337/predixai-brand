## 2. Objetivo
Eliminar representações redundantes de modos de customização e vincular o conteúdo efetivamente validado do `materializerMap` a um hash SHA-256 verificável.

## 3. Contexto
Após o hardening de IDs, respostas órfãs e conjunto exato do mapa, ainda havia múltiplas representações para a mesma intenção semântica: ausência de modo versus `use_suggested`, e omissão via `do_not_include` versus `omittedOptionalFields`. O mapa validado também não possuía hash próprio exposto na rastreabilidade.

## 4. Arquivos afetados
- `assets/js/workforce-package-effective-config.js`
- `scripts/validate_workforce_effective_config.js`
- relatório TXT desta etapa
- este Markdown
- `docs/reports_index.md`

## 5. Execução
A normalização agora produz uma forma canônica:

- `answerModes` guarda somente IDs em `edit_suggested`;
- `answers` guarda somente respostas ligadas aos IDs editados;
- `use_suggested` explícito é removido;
- `do_not_include` é convertido para `omittedOptionalFields`;
- omissões são ordenadas e não duplicadas;
- customizações semanticamente equivalentes produzem o mesmo `customizationHash`.

O `materializerMap` validado agora é reconstruído com:

- `packageId` e versão oficiais;
- conjunto exato de 19 instruções;
- instruções com Unicode NFC e espaços externos removidos;
- chaves ordenadas;
- `materializerMapHash` SHA-256 exposto em `traceability`;
- inclusão do `materializerMapHash` no cálculo do `effectiveConfigHash`.
